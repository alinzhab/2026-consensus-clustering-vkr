# Ограничения работы и направления развития

Раздел готовится к включению в текст ВКР. Каждое ограничение
формулируется явно, **сопровождается числами из реальных результатов**
(`results/analysis_full_suite.tsv`, `results/full_benchmark.tsv`,
`results/hierarchical_linkage_full_suite.tsv`,
`results/statistical_analysis.md`) и закрывается планом верификации:
скрипт в `experiments/`, который комиссия может запустить.

Цель раздела — не «посыпать голову пеплом», а показать осознанность
методологических вопросов. Это всегда зачитывается рецензентом в плюс.

## 1. Data leakage в `DEFAULT_PARAMS`

### Что именно

В `consensus_lab/sdgca_modified.py` определён словарь `DEFAULT_PARAMS`,
содержащий индивидуальные значения `(lambda, eta, theta, diffusion_time)`
для **9 датасетов**: `Ecoli`, `GLIOMA`, `Aggregation`, `Lung`, `BBC`,
`orlraws10P`, `densired_compact_hard`, `densired_stretched_hard`,
`densired_mix_hard`. На вход алгоритм для известного датасета получает
параметры, специально подобранные к его структуре.

Базовый `sdgca` при этом использует общие дефолты `(lambda=0.09,
eta=0.75, theta=0.65)` для всех датасетов. Сравнение «sdgca_modified vs
sdgca» при таком протоколе нечестное: одна сторона имеет per-dataset
гиперпараметры, другая — нет. Это классический случай data leakage в
метаданных протокола эксперимента.

### Эмпирическое подтверждение, что вопрос реален

Из `results/statistical_analysis.md` (см. раздел Friedman / Wilcoxon-Holm):
парный Вилкоксон с поправкой Холма не отвергает гипотезу равенства
`sdgca_modified` и `sdgca` (raw p ≈ 0.84, holm p = 1.0) — то есть с
учётом подбора параметров модификация **не превосходит** базовую версию.
Это уже намекает, что без подбора она тем более проиграет.

### План закрытия

```powershell
.\.venv\Scripts\Activate.ps1
python experiments\run_default_vs_tuned.py --runs 30 --m 30
```

Скрипт прогоняет каждый из 9 датасетов из `DEFAULT_PARAMS` дважды:
с подобранными параметрами и с общими дефолтами. На выходе —
`results/default_vs_tuned.tsv` и таблица «Δ NMI = tuned − generic».
Если средняя Δ значимо положительна (Wilcoxon p < 0.05) — это нужно
явно зафиксировать в тексте ВКР как «модификация требует подбора
гиперпараметров под датасет, см. таблицу X». Если Δ незначима —
исследование становится сильнее: модификация работает и без подгонки.

### Альтернатива: nested cross-validation

В тексте обоснованно сослаться на nested CV (leave-one-dataset-out для
подбора `(lambda, eta, theta)` на k−1 датасетах и валидации на k-м) как
на правильную процедуру для будущих работ. В текущей ВКР она не
проведена из-за вычислительной стоимости (см. п. 3 ниже).

## 2. Малая выборка запусков

### Что именно

Все основные эксперименты выполнены с `runs=5` (`results/full_benchmark.tsv`,
`results/hierarchical_linkage_full_suite.tsv`). При типичном `nmi_std`
порядка `0.01-0.05` стандартная ошибка среднего по 5 запускам составляет
`0.005-0.025`, что сопоставимо с разностью метрик между алгоритмами на
многих датасетах.

### Численный эффект на статистическую мощность

Объединённый анализ из `results/statistical_analysis.md` на 8 датасетах
с полным покрытием 4 алгоритмов даёт **Friedman p ≈ 0.70** — то есть на
имеющейся выборке гипотеза «все 4 алгоритма равны» **не отвергается**.
Средние ранги: `hierarchical_weighted = 2.06`, `hierarchical_baseline =
2.44`, `sdgca = 2.75`, `sdgca_modified = 2.75`. Критическое расстояние
Неменьи на этом наборе CD ≈ 1.66 — все 4 алгоритма попадают в зону
неотличимости от лидера.

При этом на расширенной выборке из `hierarchical_linkage_full_suite.tsv`
(57 пар dataset × linkage с runs=5) Wilcoxon-Holm для пары
`HC weighted vs HC baseline` даёт **p = 0.003**, гипотеза равенства
**отвергается**, среднее улучшение `Δ NMI = +0.027`. Это показывает,
что причина незначимости в Friedman-анализе — недостаточный N (8
датасетов слишком мало), а не отсутствие реального эффекта.

### План закрытия

```powershell
.\.venv\Scripts\Activate.ps1
python experiments\run_runs_stability.py --algorithm hierarchical_weighted --dataset Ecoli --max-runs 50
python experiments\run_runs_stability.py --algorithm sdgca_modified --dataset Lung --max-runs 50
```

На выходе — `results/runs_stability.tsv` и
`results/plots/08_runs_stability.png`. Цель — показать, как сужается
bootstrap-CI 95% при росте `runs`. Кандидатная рекомендация для
основной таблицы — `runs ≥ 20` (это сжимает CI типичной метрики до
ширины 0.02-0.03 NMI и делает разности 0.02 различимыми).

### Дополнительно: независимые seed

Текущая практика — фиксированный `seed=19`, варьируется только
sub-sampling из пула. Желательно повторить топ-эксперименты с 5
независимыми seed (19, 23, 29, 31, 37) и расширенным `runs`.

## 3. Квадратичная сложность по числу объектов

### Что именно

Все 4 алгоритма строят матрицы `n × n`: co-association, NWCA, S, D.
Память — `O(n² · sizeof(float64))`. На n = 5000 это уже 200 МБ только
под одну матрицу. Время — формально `O(n²)` для построения и
`O(n³ log n)` для иерархической агрегации в худшем случае.

### Эмпирическая оценка из реальных запусков

По `results/plots/05_complexity_scatter.png` (объединение всех TSV с
полем `seconds`):

- HC baseline / HC weighted: подгонка степенного закона t ≈ a · n^b
  даёт **b ≈ 1.80** на диапазоне n ∈ [50; 4000]. Это **меньше 2**, потому
  что в этом диапазоне доминирует время построения базовых
  кластеризаций (k-means с линейной по n стоимостью), а не построение
  consensus-матрицы.
- SDGCA / SDGCA modified: точки в правом верхнем углу — на n ≈ 3500
  алгоритм работает 3000-60000 секунд, что в 100-1000 раз дороже HC.
  ADMM-итерации `optimize_sdgca` решают `n × n` линейную систему на
  каждой итерации, и `expm(-t · L)` для модификации добавляет
  ещё один `O(n³)`-вызов.

### План закрытия

```powershell
.\.venv\Scripts\Activate.ps1
python experiments\run_complexity_benchmark.py --sizes 300 600 1000 1500 2000 3000
```

На фиксированных `m=20`, `K=4`, `d=4`, `runs=3` — изолируем влияние
исключительно `n`. Подгонка показателя `b` для каждого алгоритма
отдельно на одинаковых датасетах. В текст ВКР включить полученное `b`
и оценку памяти; если для SDGCA получится `b > 2.5`, это надо
зафиксировать как практическое ограничение масштаба.

### Перспективы оптимизации

- Sparse co-association через k ближайших соседей вместо полной
  матрицы: `O(n · k)` память.
- Anchor-based аппроксимация: представление матриц через `m × m`
  опорных точек (`m ≪ n`).
- Замена `expm` на спектральное усечение: `H ≈ U_k diag(exp(-t λ_k)) U_k^T`
  при `k ≪ n`.
- Численно — переход на `float32` и блочное хранение.

Эти направления — отдельные перспективные работы, в текущей ВКР не
реализуются.

## 4. Отсутствие сравнения с внешними реализациями

### Что именно

Все 4 алгоритма реализованы в проекте самим автором. Каталог с ядром
называется `consensus_lab/` — изначально он назывался `python_port/`,
что намекало на порт первоисточника и потенциально создавало вопрос об
авторстве кода. Имя каталога переименовано, и в тексте ВКР каждая
функция должна сопровождаться явной пометкой «реализация по статье X»
либо «авторская модификация». В качестве эталона сравнения используется
только собственный baseline (`hierarchical_baseline`). Это не позволяет
утверждать, что качество реализации соответствует заявленному в литературе.

### Эталоны для сравнения

- **OpenEnsembles** (Ronan et al., 2018) — ансамблевые методы кластеризации,
  включая иерархический консенсус, доступен через PyPI.
- **ClusterEnsembles** (Strehl & Ghosh) — классические CSPA / HGPA / MCLA.
- **scikit-learn** `cluster.SpectralClustering` на co-association
  matrix — простой baseline вне ансамбля.
- **LWEA / LWGP / ECGC** — современные методы, не рассмотренные в работе.

### План закрытия

Добавить в `experiments/` скрипт `run_external_baselines.py`,
прогоняющий те же датасеты через один-два внешних пакета и
сохраняющий метрики в общий формат `results/*.tsv`. Включить в
сравнительную таблицу. Из всех ансамблевых пакетов наиболее
доступный — OpenEnsembles (`pip install OpenEnsembles`); приоритет
ему.

В рамках текущей ВКР рекомендуется хотя бы один внешний baseline
(MCLA или CSPA через ClusterEnsembles): иначе остаётся уязвимость
«может быть, ваш собственный baseline просто плохо реализован».

## 5. Узкий набор датасетов и доменов

8 датасетов с полным покрытием — нижняя граница для критерия
Фридмана (разница рангов в 0.5 при k = 4 не достигает CD = 1.66).
Расширение до 15-20 датасетов с явным расслоением «реальные / синтетические /
плотностные / высокоразмерные» резко повышает мощность.

Дополнительно: реальные `Ecoli`, `Lung`, `GLIOMA` имеют меньше 350
объектов — для метрик с pairwise-членом это малая выборка с высокой
дисперсией.

## 6. Номенклатура каталога ядра (устранено)

Раннее каталог ядра назывался `python_port/`, что подразумевало порт
кода с другого языка и потенциально ставил вопрос об авторстве. Каталог
переименован в `consensus_lab/` (см. раздел README «Почему каталог
называется `consensus_lab`»). Остаётся методологическая задача: в тексте
ВКР для каждой функции явно указать, является ли она реализацией по
статье (со ссылкой на первоисточник) или авторской модификацией. В
README.md рекомендуется добавить раздел «Происхождение кода» и пометить
комментариями `# Original`, `# Implementation of <ref>`, `# Modification
by author` соответствующие функции — это снимает вопрос об авторстве на
уровне репозитория.

## 7. Веб-интерфейс не оценивался отдельно

Веб-приложение упомянуто как вспомогательный артефакт. Юзабилити-
тестирование, нагрузочное тестирование, сценарии конкурентного доступа
не проводились. Это ограничивает заявления о практической применимости
системы. Если внедрение упоминается в тексте, оно должно сопровождаться
актом или хотя бы скриншотами реального использования.

## 8. Отсутствие теоретического анализа сложности

В тексте ВКР для каждого из четырёх алгоритмов следует выписать
оценку асимптотической вычислительной и пространственной сложности
с пояснением вкладов. Текущий `limitations.md` указывает эмпирические
показатели степени; они должны соотноситься с теоретическими.

Кандидатная таблица для текста:

| Алгоритм                | Время (доминирующий член)             | Память |
|-------------------------|---------------------------------------|--------|
| HC baseline             | O(m·n²) построение CA + O(n²) clust   | O(n²)  |
| HC weighted             | O(m²·n²) — парные согласия            | O(n²)  |
| SDGCA                   | O(it·n³) — линейная система в ADMM    | O(n²)  |
| SDGCA modified          | то же + O(n³) для expm(L)             | O(n²)  |

(Числа — ориентировочные; в тексте ВКР подкрепить ссылками на конкретные
строки реализации.)

## Сводка по направлениям развития

1. Полноценная nested cross-validation для гиперпараметров SDGCA.
2. Расширение бенчмарка до 15-20 датасетов в 4 доменах.
3. Сравнение с OpenEnsembles / ClusterEnsembles на тех же данных.
4. Sparse / anchor-based варианты SDGCA для n ≥ 10000.
5. Спектральная аппроксимация `expm(L)` через первые k собственных пар.
6. Адаптивный подбор `(lambda, eta, theta)` по характеристикам ансамбля.
7. Расширение веб-системы: очередь задач, авторизация, история запусков.
